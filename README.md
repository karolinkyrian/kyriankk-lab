# Лабораторна робота №1. Фронтенд  
## Варіант №9  

**Тема:** Заявки в технічну підтримку для студентів  
**Ключова сутність:** Заявка  

## Опис проєкту

Веб-застосунок для створення та керування заявками в технічну підтримку студентів. Реалізовано CRUD-операції, пошук, сортування та збереження даних у браузері.

## Функціонал

- Створення нових заявок  
- Відображення заявок у вигляді таблиці  
- Редагування заявок  
- Видалення заявок  
- Вибір статусу та пріоритету заявки  
- Валідація обов’язкових полів форми  
- Пошук по заявках  
- Сортування по колонках (тема, статус, пріоритет, автор)  
- Перемикання сортування (включення/вимикання)  
- Підсвітка активної колонки сортування  
- Захист від дублювання заявок  
- Збереження даних у localStorage  
- Автоматичне відновлення даних після перезавантаження сторінки
  
## Запуск проєкту

Запуск локального сервера:
python -m http.server 8080

Відкриття у браузері:
http://localhost:8080/frontend/

## Схема бази даних (взаємопов'язані таблиці з ON DELETE CASCADE)

Users: (id, email UNIQUE, name)

Tickets: (id, userId FK, subject, description, status, priority)

TicketMessages: (id, ticketId FK, userId FK, message)

## Реалізовані можливості

- Автоматична ініціалізація бази даних та створення таблиць.
- Механізм Seed для автоматичного наповнення бази тестовими даними (3 користувачі, стартові заявки та повідомлення).
- Тришарова архітектура (робота з БД винесена в окремий шар Repositories).
- Складні SQL-запити: використання JOIN для об'єднання даних (ендпойнт /api/tickets/full повертає заявки разом з іменами авторів).
- Обмеження на рівні бази даних: CHECK для статусів (new, in_progress, done) та пріоритетів (low, medium, high), а також індекс idx_tickets_status.

## Примітки щодо безпеки (SQL Injection)

Згідно з навчальною метою лабораторної роботи, фільтрація та сортування в деяких запитах реалізовані через конкатенацію рядків. Це робить систему вразливою до SQL Injection. Наприклад, передача ' OR 1=1; -- у параметри пошуку дозволяє обійти логіку вибірки. Вразливості будуть усунені в наступних роботах за допомогою параметризованих запитів.

## Приклади запитів (cURL)

1. Отримати всі заявки разом з іменами авторів (Складний запит з JOIN - 200 OK):
   ```bash
   curl -i http://localhost:3000/api/tickets/full

2. Створити нову заявку в базу даних (201 Created):
   ```bash
   curl -i -X POST http://localhost:3000/api/tickets -H "Content-Type: application/json" -d "{\"userId\":1,\"subject\":\"Не працює Wi-Fi\",\"description\":\"У гуртожитку немає мережі\",\"status\":\"new\",\"priority\":\"high\"}"

3. Отримати заявки з фільтрацією та сортуванням (200 OK):
   ```bash
   curl -i "http://localhost:3000/api/tickets?sort=priority&order=DESC"

4. Видалити заявку та каскадно всі її повідомлення (200 OK):
   ```bash
   curl -i -X DELETE http://localhost:3000/api/tickets/1

## Реалізовані можливості
**Бекенд та API (Версія v1)**
- Версійність: всі ендпоінти використовують префікс /api/v1/.
- CORS: налаштовано суворий доступ лише для origin фронтенду (без використання *). Дозволено методи: GET, POST, PUT, DELETE.
- Централізована обробка помилок: повернення коректних HTTP-кодів (400, 404, 500) та валідація вхідних DTO на рівні сервера.
- Принцип стабільності: Реалізовано принцип «не ламати формат» DTO для забезпечення сумісності з клієнтом.

**Фронтенд та інтеграція**
- Модульна архітектура: код розділено на модулі (apiClient.js, ui.js, config.js, main.js) з використанням type="module".
- Тонкий клієнт API: створено уніфікований сервіс apiClient для виконання запитів, обробки response.ok та парсингу JSON в одному місці.
- Стани завантаження: реалізовано візуальні стани «Завантаження…», «Немає даних» та «Помилка завантаження».
- Обробка сценаріїв: коректна реакція UI на недоступність сервера (500) або помилки валідації (400) через сповіщення в блоці notice.

## Сценарії для перевірки (QA)
1. GET (Успіх): завантаження списку заявок при старті сторінки.
2. POST (Створення): заповнення форми та збереження. Рядок з'являється в таблиці без перезавантаження сторінки.
3. PUT (Редагування): натискання «Редагувати», зміна полів, збереження. Рядок підсвічується тінню під час редагування.
4. DELETE (Видалення): видалення заявки з каскадним оновленням таблиці.
5. CORS/Error (Помилка): спроба запиту з недозволеного домену або при вимкненому бекенді — виводиться червоне повідомлення про помилку.

## Специфікація DTO заявки:

Ідентифікація: кожна заявка має числовий id, що генерується автоматично базою даних.

Валідація тексту: поле subject є обов'язковим і має проходити перевірку на мінімальну довжину (3 символи).

Стабільність контрактів: поля status та priority використовують обмежений набір значень (Enum), що забезпечує коректне відображення в UI.

Зв'язок: поле userName передається як рядок для спрощення відображення без додаткових запитів до таблиці користувачів.
